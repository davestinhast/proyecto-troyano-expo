LIBRECRI — Guía para Entender el Proyecto

Una explicación sencilla de cómo funciona este programa de acceso remoto, para quienes no trabajan con seguridad todos los días.

Año 2026

Resumen Rápido

LIBRECRI es un programa que se instala en secreto en una computadora con Windows. Desde otra computadora (la nuestra, en Kali Linux), podemos darle órdenes: ver la pantalla, escuchar el micrófono, ver qué teclas presiona el usuario, y más. Todo se comunica por internet usando el protocolo HTTP, igual que cuando abres una página web, y los mensajes van cifrados para que nadie más los entienda. El proyecto se hizo con fines educativos, para aprender cómo funcionan estos ataques y así poder defenderse mejor de ellos.

⚠ Todo lo que se muestra aquí se probó únicamente en computadoras de laboratorio que nos pertenecen. Usar este programa en computadoras ajenas sin permiso es un delito.
01

¿Qué es un Troyano?

Un troyano (o troyano de acceso remoto, RAT por sus siglas en inglés) es un programa que se instala en una computadora haciéndose pasar por otra cosa. Una vez dentro, le da al atacante control total sobre esa máquina desde lejos, sin que el dueño de la computadora se dé cuenta.

Analogía

Imagina que el caballo de Troya de la mitología griega es un archivo llamado actualizacion.exe. El usuario lo abre creyendo que es una actualización real. Dentro hay soldados (el troyano) que abren la puerta de la ciudad (dan acceso al atacante) desde adentro.

Los troyanos reales que usan los hackers en el mundo funcionan exactamente igual que LIBRECRI. Familias famosas como Cobalt Strike, Meterpreter o njRAT usan las mismas ideas que aplicamos en este proyecto. La diferencia es que LIBRECRI lo construimos nosotros desde cero para entender cada parte.

En LIBRECRI, el troyano se llama implant (el que se instala en la víctima) y el programa que lo controla se llama servidor C2 o simplemente el C2 (Command and Control — control y comando).

  • generador.py: es el script de Python que fabricamos. Cuando lo ejecutamos, nos pregunta la dirección IP y el puerto, y genera automáticamente tanto el troyano (implant.exe) como el servidor (c2_server.py) con claves únicas de seguridad.
  • implant.exe: el programa que corre en la computadora víctima (en este caso, una máquina Windows de laboratorio). No abre ninguna ventana, corre en silencio en segundo plano.
  • c2_server.py: el programa que corremos en nuestra máquina Kali. Nos muestra una consola donde escribimos órdenes y vemos los resultados.

El ciclo de comunicación

El implant no espera que el servidor lo llame. Cada 850 milisegundos (menos de un segundo), el implant es el que pregunta al servidor: "¿Hay alguna orden para mí?" Esto se llama polling. Si no hay nada, el servidor responde vacío y el implant vuelve a esperar. Si hay una orden, el implant la ejecuta y manda el resultado de vuelta.

01 El implant en la PC víctima pregunta cada 850ms: "¿Tienes una orden para mí?" usando una petición HTTP normal, igual que cuando el navegador pide una página web.
02 El servidor C2 responde con la orden cifrada (o sin nada si no hay orden pendiente).
03 El implant descifra la orden, la ejecuta en la PC víctima (captura pantalla, ejecuta un comando, etc.) y guarda el resultado.
04 El implant manda el resultado de vuelta al servidor, cifrado. El servidor lo descifra y nos lo muestra en pantalla.
Analogía

Es como un empleado que cada pocos segundos revisa su buzón de correo para ver si el jefe le mandó algo. Si hay una carta (una orden), la lee, hace lo que dice, y manda la respuesta de vuelta. Si el buzón está vacío, simplemente vuelve a esperar.

03

¿Cómo le Damos Órdenes al Troyano?

Cuando el servidor C2 detecta que el implant se conectó, nos muestra un símbolo de espera donde podemos escribir órdenes en texto. Es parecido a usar la terminal o el símbolo del sistema (cmd), pero los comandos se ejecutan en la computadora de la víctima, no en la nuestra.

Por ejemplo, si escribimos ls, el servidor manda esa orden al implant, el implant ejecuta el equivalente en Windows (dir), y nos manda la lista de archivos. Si escribimos screenshot, el implant toma una foto de la pantalla y nos la manda como imagen.

Todo el texto que mandamos y recibimos viaja cifrado. Alguien que intercepte el tráfico de red solo vería una serie de bytes sin sentido, no el texto real de las órdenes ni los resultados.

04

¿Cómo se Esconde el Implant?

Los antivirus buscan programas maliciosos de varias formas. LIBRECRI usa varias técnicas para hacer esa búsqueda más difícil:

1. Las herramientas que usa están escondidas dentro del programa

Cuando un antivirus analiza un programa, una de las primeras cosas que revisa es: ¿qué funciones del sistema operativo usa? Si ve que el programa llama a funciones como "crear un proceso nuevo" o "conectarse a internet", levanta una alerta. LIBRECRI guarda los nombres de esas funciones como texto mezclado con una clave secreta. El programa los descifra solo cuando los necesita, justo en el momento de usarlos.

Analogía

Es como si en vez de llevar un cuchillo en el cinturón (donde todos lo ven), lo llevaras desmontado en una maleta con las piezas marcadas con un código que solo tú entiendes. Para el guardia de seguridad, parece equipaje normal.

2. Usamos una clave diferente cada vez que compilamos

Cada vez que el generador crea un nuevo implant, genera una clave secreta al azar. Eso significa que dos implants creados en días diferentes se ven completamente diferentes en bytes, aunque hagan lo mismo. Los antivirus buscan "firmas" (patrones de bytes conocidos), y si los bytes cambian, la firma ya no coincide.

3. El programa tiene importaciones de "mentira"

Para que no llame la atención que el programa tiene la lista de herramientas vacía (lo cual también es sospechoso), mostramos una lista falsa de funciones que usa: cosas como herramientas de sonido o interfaz gráfica que no levantan alarmas. Las herramientas reales están escondidas como explicamos antes.

4. No abre ninguna ventana

El implant corre en el fondo, sin mostrar ninguna ventana de consola ni interfaz. Para el usuario, es como si no existiera.

5. El tráfico de red parece normal

La comunicación usa HTTP, el mismo protocolo de las páginas web. Incluso el encabezado que manda imita al de un navegador Chrome. Para alguien mirando el tráfico de red sin profundizar, parece una visita normal a un sitio web.

05

¿Qué Puede Hacer el Implant?

LIBRECRI incluye cuatro módulos de captura además de control completo del sistema de archivos y línea de comandos:

5.1 Captura de Pantalla

Con el comando screenshot, el implant le pide a Windows que copie todo lo que se está mostrando en la pantalla en ese momento y lo guarde como imagen. Luego esa imagen viaja al servidor cifrada y convertida en texto (en formato Base64, que es una manera de convertir archivos en texto plano para poder mandarlos por internet). El servidor la vuelve a convertir en imagen y la guarda en nuestro escritorio.

Un detalle técnico: las imágenes de pantalla llena pueden ser bastante grandes (más de 1 MB de datos). Por eso tuvimos que ampliar el tamaño máximo del mensaje que el implant puede manejar a 4 MB. Antes era de 128 KB y la imagen llegaba cortada a la mitad, lo que la dejaba dañada.

5.2 Fotografía con la Cámara Web

Con webcam_snap, usamos una herramienta de Windows llamada avicap32.dll que permite controlar cámaras de video. El implant le pide al sistema que tome una foto con la primera cámara conectada, espera un segundo y medio para que la cámara se estabilice, y luego guarda la imagen y la manda de vuelta. La foto llega a nuestro escritorio como archivo .bmp.

5.3 Registro de Teclas (Keylogger)

Con keyscan_start, el implant activa un "gancho" en Windows. Windows tiene un sistema donde un programa puede pedir que se le avise de cada tecla que presiona el usuario en toda la computadora, sin importar en qué ventana esté. El implant activa este aviso, va guardando las teclas en un registro interno, y cuando le pedimos con keyscan_dump, nos manda todo lo que se tecleó desde que activamos el gancho.

Analogía

Es como poner un micrófono en la máquina de escribir. Cada golpe de tecla queda grabado, sin importar si estás en el navegador, en el procesador de texto o en el chat.

5.4 Grabación de Micrófono

Con record_mic -d 10 (por ejemplo, para grabar 10 segundos), el implant usa las herramientas de audio de Windows (winmm.dll) para grabar lo que capta el micrófono. Al terminar, construye manualmente un archivo de audio WAV y lo manda al servidor. El archivo llega a nuestro escritorio listo para reproducir.

Este fue el módulo que más problemas dio. El comando que se le manda al implant para iniciar la grabación es bastante largo (más de 6,000 caracteres), y el implant tenía un límite de 4,096 caracteres por mensaje. El comando llegaba cortado y fallaba en silencio. La solución fue ampliar ese límite a 24,000 caracteres.

06

¿Por Qué No lo Detectan los Antivirus?

Los antivirus modernos buscan programas maliciosos de dos formas principales:

  • Análisis estático: revisan el archivo sin ejecutarlo, buscando patrones conocidos (firmas) o cosas sospechosas como listas de herramientas de red y control de procesos.
  • Análisis dinámico: ejecutan el programa en un entorno controlado y miran qué hace.

Las técnicas que usamos (claves aleatorias por compilación, esconder las herramientas en texto cifrado, lista de importaciones falsa) van dirigidas principalmente a engañar el análisis estático. Frente a un antivirus moderno con análisis de comportamiento avanzado, el implant terminaría siendo detectado con el tiempo. Eso está bien para los propósitos del proyecto: lo importante era aprender cómo funciona el ataque, no crear algo indetectable.

ℹ Los programas de seguridad empresarial modernos como CrowdStrike o SentinelOne sí detectarían el keylogger y el uso de PowerShell para compilar código. La protección existe y funciona. Por eso se recalca la importancia de mantener los sistemas actualizados.

¿Cómo nos Defendemos de Algo Así?

Conocer cómo funciona un ataque es la mejor forma de defenderse. Estas son las señales de alerta que deja LIBRECRI y las formas de bloquearlo:

Señales de alerta (indicadores de compromiso)

  • Conexiones HTTP repetidas cada un segundo aproximadamente al mismo servidor externo, sin que el usuario haya abierto nada.
  • Un programa corriendo sin ninguna ventana visible que hace conexiones a internet.
  • Presencia de un registro en Windows que arranca un programa desconocido al iniciar la sesión.
  • Un programa que activa el gancho de teclado de Windows sin ser una aplicación de accesibilidad o seguridad reconocida.

Formas de protegerse

  • Mantener el antivirus y el sistema actualizados: las bases de datos de firmas se actualizan constantemente y eventualmente reconocen variantes nuevas.
  • No ejecutar archivos de fuentes desconocidas: el implant necesita que alguien lo ejecute. La ingeniería social (engañar al usuario para que lo abra) es el punto de entrada más común.
  • Usar soluciones de seguridad empresarial (EDR): programas como Microsoft Defender for Endpoint, CrowdStrike o SentinelOne detectan comportamientos sospechosos aunque no reconozcan el archivo exacto.
  • Monitorear el tráfico de red: herramientas como Suricata o Zeek pueden detectar patrones de polling regular (conexiones repetitivas y periódicas) que son inusuales en tráfico normal.
  • Restringir qué programas pueden correr (lista blanca de aplicaciones): Windows tiene una función llamada WDAC que solo permite ejecutar programas autorizados. Un archivo descargado de internet no estaría en esa lista.
09

¿Cómo lo Hice?

Esta sección explica el proceso de construcción del proyecto, paso a paso, en palabras simples. Sirve también como respuesta directa si alguien pregunta durante la presentación.

Punto de partida

Empecé estudiando cómo los troyanos reales se comunican con su servidor. La mayoría usa HTTP porque es el protocolo que casi nunca bloquean los firewalls (si lo bloquearan, no habría internet). Una vez entendido eso, el proyecto se dividió en tres partes: el troyano, el servidor y el generador que los fabrica juntos.

El troyano (implant.exe)

Lo escribí en C puro, que es el lenguaje más cercano al sistema operativo y produce archivos .exe más pequeños y difíciles de analizar que Java o Python. Usé las herramientas del sistema de Windows directamente: funciones para conectarse a internet (ws2_32.dll), para ejecutar comandos (kernel32.dll), para el keylogger (user32.dll) y para el audio (winmm.dll).

Para compilarlo en Kali Linux y que funcione en Windows, usé mingw-w64, que es un compilador cruzado: compila para Windows desde Linux.

El servidor (c2_server.py)

Lo escribí en Python. Python trae incluida una librería para hacer servidores web sencillos (http.server), así que no necesité instalar nada externo. El servidor muestra una consola interactiva donde escribo las órdenes, las cifra, las manda al implant, recibe la respuesta, la descifra y la muestra.

El generador (generador.py)

Es el script principal. Cuando lo ejecuto, genera las claves secretas al azar, las mete en el código fuente del troyano, compila el troyano y genera el servidor. Es como una fábrica: le pongo la dirección IP y el puerto, y me entrega el par implant+servidor listos para usar, configurados juntos con las mismas claves.

Las técnicas de ocultamiento

Las aprendí estudiando documentación técnica de Microsoft (cómo funciona el Import Address Table de los archivos .exe de Windows) y libros como Practical Malware Analysis. La idea de guardar los nombres de funciones cifrados con XOR es una técnica clásica que aparece en malware real. La apliqué para entender por qué funciona y cómo se detectaría.

Los módulos de captura (pantalla, cámara, micrófono)

Para captura de pantalla, usé PowerShell con las librerías de Windows ya instaladas (System.Drawing). Para la cámara y el micrófono, el truco fue escribir el código en C# directamente en el script de PowerShell y pedirle que lo compile en el momento de ejecutarse. Así no hay archivos extra en la PC víctima: el código se compila en memoria, corre, y desaparece.

✓ Resumen para responder en presentación: "Usé Python para el generador y el servidor, C para el troyano, y PowerShell con C# inline para los módulos de captura. Todo se compila y genera automáticamente con un solo script."
10

Conclusión

LIBRECRI logró todos los objetivos del proyecto. Pudimos controlar una PC con Windows desde Kali Linux: ver su pantalla, escuchar su micrófono, fotografiar con su cámara y registrar sus teclas. Todo a través de una conexión HTTP normal, con mensajes cifrados que no llaman la atención.

Lo más valioso del proyecto no son los módulos que funcionan, sino lo que aprendimos al construirlos:

  • Un troyano no necesita ser complicado para ser funcional. Las herramientas que necesita ya están instaladas en Windows.
  • El punto más débil de cualquier sistema de seguridad no es el antivirus: es el usuario que ejecuta un archivo sin verificar su origen.
  • Saber cómo se construye un ataque permite escribir defensas mucho más precisas: reglas de red, configuraciones de seguridad y alertas que van al punto.
  • Las soluciones de seguridad modernas (EDR, análisis de comportamiento) sí funcionan. Los módulos que más llaman la atención, como el registro de teclas y la compilación de C# en tiempo real, serían detectados por esos sistemas.

Preguntas Frecuentes

Respuestas listas para las preguntas más comunes durante una presentación:

¿Por qué usaste HTTP y no otro protocolo?
Porque HTTP es el protocolo de las páginas web. Casi ningún firewall lo bloquea porque cortarlo implicaría quedarse sin internet. Es el canal más conveniente para un atacante.
¿Qué es el cifrado XOR que mencionas?
XOR es una operación matemática que mezcla cada byte de un mensaje con un número de una clave secreta. Para descifrar solo hace falta aplicar la misma operación con la misma clave. No es cifrado de alta seguridad, pero es suficiente para que el texto de las órdenes no sea legible a simple vista en el tráfico de red.
¿Esto podría usarse fuera del laboratorio?
Técnicamente sí, pero legalmente no. Usar este programa en computadoras sin autorización explícita del dueño es un delito en prácticamente todos los países. El proyecto existe solo en un entorno de laboratorio controlado, con máquinas que nos pertenecen.
¿Cómo lo detectaría un antivirus moderno?
Un antivirus con análisis de comportamiento detectaría el gancho de teclado (porque pide acceso a las teclas de todo el sistema) y el uso de PowerShell para compilar código en tiempo real (porque es una técnica que usa mucho el malware moderno). Las técnicas de ocultamiento que aplicamos van contra los antivirus que solo buscan patrones en los bytes del archivo, no los que analizan qué hace el programa mientras corre.
¿Cuánto tiempo te llevó hacerlo?
El proyecto se construyó de forma incremental a lo largo del curso. Empecé con la comunicación básica (mandar y recibir un texto), luego agregué el cifrado, después los módulos de captura uno por uno, y al final resolví los errores de tamaño de buffer que impedían que los módulos de cámara y micrófono funcionaran correctamente.
¿Por qué programaste el troyano en C y no en Python?
Python necesita que la máquina víctima tenga Python instalado, o el archivo sería muy grande (porque habría que empaquetar todo el intérprete de Python adentro). En C el resultado es un ejecutable nativo de Windows de pocos kilobytes que no necesita nada extra instalado.
¿Qué fue lo más difícil del proyecto?
El módulo de micrófono. El comando que se le manda al implant para iniciar la grabación es muy largo (más de 6,000 caracteres) y el implant tenía un límite de 4,096 caracteres. El comando llegaba cortado, PowerShell recibía texto inválido, y fallaba en silencio sin dar ningún error claro. Encontrar ese límite y ajustarlo fue lo que más tiempo llevó.
