Resumen Ejecutivo

LIBRECRI es un framework ofensivo educativo que demuestra los principios de un Remote Access Trojan (RAT) construido desde cero. El proyecto comprende un generador Python que produce pares únicos de ejecutable Windows (implant C) y servidor de comando y control (C2) en Python, comunicados mediante polling HTTP con cifrado XOR simétrico por compilación. Se implementaron técnicas de evasión de detección estática y dinámica con fines de aprendizaje sobre amenazas reales.

01

Introducción y Objetivos

El presente proyecto tiene como finalidad demostrar, en un entorno controlado y con propósitos estrictamente académicos, cómo funciona internamente un troyano de acceso remoto moderno. Comprender la mecánica de estas herramientas es fundamental para cualquier profesional de seguridad ofensiva y defensiva: no es posible detectar ni mitigar lo que no se comprende.

LIBRECRI (acrónimo de Libre C Remote Implant) reimplementa desde cero funcionalidades que aparecen en familias de malware reales como Cobalt Strike Beacon, Meterpreter y njRAT, pero en un contexto transparente y auditable donde cada decisión de diseño está documentada.

Objetivos específicos

  • Implementar un canal de comunicación C2 basado en HTTP polling sobre un socket TCP ordinario.
  • Aplicar cifrado XOR por compilación que haga cada binario diferente en bytes, evitando firmas estáticas triviales.
  • Demostrar técnicas de resolución dinámica de API para ocultar las importaciones del ejecutable (IAT camouflage).
  • Implementar módulos de post-explotación: keylogger de bajo nivel, captura de pantalla, fotografía por webcam y grabación de micrófono.
  • Analizar los indicadores de compromiso (IOC) que genera el sistema y las contramedidas defensivas aplicables.
ℹ Todo el código fue desarrollado y ejecutado únicamente en entornos de laboratorio aislados bajo supervisión académica. El uso de estas técnicas fuera de un contexto autorizado es ilegal en la mayoría de jurisdicciones.

Arquitectura del Sistema

El sistema se compone de tres elementos que trabajan en conjunto. El generador produce todos los artefactos necesarios para una operación; el implant corre en la máquina víctima; el servidor C2 corre en la máquina del operador.

2.1 Generador (generador.py)

El generador es el núcleo del framework. Al ejecutarse, solicita los parámetros de la operación (LHOST, LPORT, nombre del ejecutable) y realiza las siguientes tareas:

  • Genera las claves criptográficas únicas de la compilación: FNK (1 byte) y PK (16 bytes).
  • Construye el código fuente C del implant embebiendo dichas claves y el LHOST/LPORT cifrados.
  • Compila el implant con mingw-w64 (cross-compilation para Windows en Kali).
  • Genera el c2_server.py correspondiente a ese par de claves.
  • Sirve el ejecutable por HTTP en el puerto 8080 para facilitar la entrega.

2.2 Implant C (_lbr_http.c)

Escrito en C puro orientado a Windows 7+ (aunque se testea en Windows 10/11). Usa exclusivamente resolución dinámica de API y no tiene dependencias de librerías de importación en su IAT visible, excepto algunos decoys. El implant:

  • Se instala con un constructor __attribute__((constructor)) que corre antes de WinMain.
  • Verifica la presencia de un mutex global para evitar múltiples instancias.
  • Lanza el hilo de C2 (_c2) y espera indefinidamente.

2.3 Servidor C2 (c2_server.py)

Servidor HTTP mínimo implementado sobre BaseHTTPRequestHandler. Expone dos endpoints y una consola interactiva estilo Meterpreter para el operador. Cada par implant/servidor es indisoluble: un servidor generado con claves distintas rechaza silenciosamente al implant incorrecto.

03

Protocolo de Comunicación HTTP

LIBRECRI usa HTTP/1.0 polling sobre TCP como canal encubierto. La elección de HTTP sobre sockets raw (como en RATs clásicos tipo njRAT sobre TCP/UDP) se justifica por dos razones: es el protocolo más presente en cualquier red corporativa o doméstica y traversa la mayoría de firewalls sin requerir reglas especiales.

Flujo de polling

01 El implant despierta cada 850ms, abre una conexión TCP a LHOST:LPORT y envía GET /cmd HTTP/1.0 con cabeceras de User-Agent y X-MID (identificador de máquina).
02 El servidor responde con Content-Length: 0 (sin comando pendiente) o con el comando cifrado en el cuerpo de la respuesta.
03 Si hay comando, el implant lo descifra (XOR con PK), lo ejecuta y captura la salida estándar y de error mediante pipes.
04 El implant cifra la salida (XOR con PK), la envía via POST /res HTTP/1.0 y cierra el socket. El servidor descifra y muestra el resultado en la consola del operador.

Identificación de máquina

Cada petición GET incluye la cabecera X-MID: <hostname> obtenida con GetComputerNameA. El servidor registra cada IP + hostname, mostrando si es conexión nueva o reconexión, y silencia el spam de reconexiones durante una sesión activa.

Handshake suave

Al establecer sesión, el servidor envía echo LIBRECRI_OK y espera la respuesta. Si no llega en 10 segundos pero el implant ha realizado 3 o más polls, la sesión se acepta igualmente. Esto cubre casos donde el buffer de respuesta llega retrasado por latencia de red.

04

Cifrado XOR por Compilación

LIBRECRI utiliza dos capas de XOR con propósitos distintos:

ClaveTamañoPropósitoGeneración
FNK 1 byte Cifra los nombres de API en el binario (Function Name Key). Evita que strings como CreateThread aparezcan literales en el ejecutable. secrets.randbelow(0xFF)
PK 16 bytes Cifra el tráfico HTTP (comandos y resultados). Aplicado en modo stream circular: p[i] ^= PK[i % 16]. secrets.token_bytes(16)

Ambas claves son generadas aleatoriamente en cada ejecución del generador y quedan embebidas como constantes en el ejecutable compilado. El servidor c2_server.py correspondiente recibe las mismas claves en texto plano dentro de su código Python.

Cifrado de strings en el binario

Todos los nombres de funciones API, rutas de registro, nombres de mutex y cadenas de red se almacenan en el ejecutable como arrays de bytes XOR'd con FNK. En tiempo de ejecución la función _xs() los descifra sobre el stack justo antes de usarlos:

// Ejemplo: "CreateMutexA" XOR 0x42 → array cifrado en .rodata
static const unsigned char _k5[] = {0x01,0x30,0x23,0x27,0x23,0x2E...};
char fn[32];
_xs(fn, _k5, 12);  // → "CreateMutexA" en el stack
gCMA = (void*)GetProcAddress(hk, fn);
⚠ El XOR de 1 byte (FNK) y 16 bytes (PK) no es criptografía segura — es ofuscación. Ante un análisis dinámico con debugger, las claves son recuperables trivialmente. El objetivo es evadir análisis estático automatizado (AV signatures), no proteger datos contra adversarios activos.
05

Resolución Dinámica de API (IAT Camouflage)

El Import Address Table (IAT) de un ejecutable Windows lista todas las DLLs y funciones que importa. Herramientas de análisis como dumpbin, Detect-It-Easy o soluciones AV/EDR inspeccionan el IAT para clasificar un binario. Si el IAT contiene CreateRemoteThread, VirtualAllocEx o WSASocket, levanta alertas inmediatas.

Técnica aplicada

LIBRECRI no importa ninguna función sospechosa en el IAT. En su lugar, carga las DLLs en tiempo de ejecución y resuelve cada función mediante GetProcAddress usando los nombres descifrados dinámicamente:

// _load_libs() — llamada desde el constructor, antes de WinMain
HMODULE hk = GetModuleHandleA("kernel32.dll");  // ya en memoria
_xs(fn, _k5, 12);                              // descifra "CreateThread"
gCTH = (void*)GetProcAddress(hk, fn);          // guarda en puntero global

Decoy IAT

Para que el IAT no aparezca vacío (lo cual también es sospechoso), el implant importa explícitamente 10 funciones inocuas de 5 DLLs distintas:

DLLFunciones importadas (decoy)
comctl32.dllInitCommonControlsEx, ImageList_Create
crypt32.dllCertOpenSystemStoreA, CertCloseStore
version.dllGetFileVersionInfoSizeA, GetFileVersionInfoA
winmm.dllwaveOutGetNumDevs, waveOutGetVolume

Estas funciones están agrupadas en una función _pad_init() marcada como dead code (nunca se invoca en tiempo real), pero el compilador las mantiene en el binario por el array de punteros _fiat[] marcado como volatile.

Módulos del Implant

6.1 Keylogger — WH_KEYBOARD_LL

El keylogger usa el hook de bajo nivel de Windows WH_KEYBOARD_LL (hook #13). Al recibir el comando KSTART, el implant lanza un hilo dedicado (_kthread) que:

  1. Instala el hook con SetWindowsHookExA(WH_KEYBOARD_LL, _kbhook, NULL, 0).
  2. Ejecuta un message loop (GetMessageA) que mantiene el hilo vivo y dispara las callbacks.

La función de callback _kbhook intercepta WM_KEYDOWN y WM_SYSKEYDOWN, mapea el virtual key code a caracteres legibles (letras, números, [ENTER], [BS], [TAB], etc.) y los acumula en un buffer circular de 32 KB.

static LRESULT CALLBACK _kbhook(int nCode, WPARAM wParam, LPARAM lParam) {
  if (nCode >= 0 && (wParam == 0x0100 || wParam == 0x0104)) {
    _KBLL *kb = (_KBLL*)lParam;
    char ch[16] = {0};
    _kmap(kb->vkCode, ch);        // VK → string legible
    // acumula en g_kbuf[g_kidx]
  }
  return gCNHEX(NULL, nCode, wParam, lParam); // pasa al siguiente hook
}

Comandos asociados: keyscan_start · keyscan_dump (vacía el buffer) · keyscan_stop

6.2 Captura de pantalla

Implementado como un one-liner de PowerShell enviado como comando al implant. PowerShell usa las APIs de System.Windows.Forms y System.Drawing:

  1. Obtiene las dimensiones de la pantalla principal con Screen.PrimaryScreen.Bounds.
  2. Crea un Bitmap y copia el escritorio con Graphics.CopyFromScreen.
  3. Serializa el Bitmap como PNG en un MemoryStream.
  4. Retorna el PNG codificado en Base64 como salida estándar.

El servidor recibe el Base64, limpia whitespace, corrige el padding y guarda el archivo PNG en ~/Desktop/screenshot_<timestamp>.png.

6.3 Cámara web (webcam_snap)

Usa la API avicap32.dll de Windows para capturar un frame de la cámara web a través de Direct Show. Se entrega como un script C# compilado en tiempo de ejecución por PowerShell via Add-Type y transmitido como Encoded Command Base64 para evitar problemas de escape en la línea de comandos:

  1. capCreateCaptureWindowA: crea una ventana de captura invisible.
  2. WM_CAP_DRIVER_CONNECT (0x040A): conecta el primer dispositivo de video.
  3. Espera 1.5s para que la cámara se inicialice.
  4. WM_CAP_GRAB_FRAME (0x043C): captura un frame.
  5. WM_CAP_FILE_SAVEDIB (0x0417): guarda como BMP en %TEMP%\wc.bmp.
  6. Lee el BMP, codifica en Base64 y lo retorna por stdout.

El resultado se guarda en ~/Desktop/webcam_<timestamp>.bmp.

6.4 Grabación de micrófono

Usa la API waveIn* de winmm.dll (Windows Multimedia) para grabación PCM sin dependencias externas. El script C# compilado en tiempo de ejecución:

  1. waveInOpen: abre el dispositivo de audio con formato PCM 16kHz, mono, 16 bits.
  2. Asigna un buffer para N * 32000 bytes (2 bytes/muestra × 16000 muestras/s × N segundos).
  3. waveInStart + Thread.Sleep(N*1000 + 300ms): graba durante N segundos.
  4. waveInStop: detiene la grabación.
  5. Construye manualmente el encabezado WAV RIFF/PCM y retorna el archivo completo en Base64.

El resultado se guarda en ~/Desktop/mic_<timestamp>.wav.

✓ El comando del micrófono (incluido el script C# codificado en Base64 UTF-16LE) tiene una longitud de ~6,272 bytes. Esto motivó aumentar el buffer de recepción del implant (CBUF) de 4 KB a 24 KB.

Técnicas de Evasión Implementadas

TécnicaDescripciónContramedida que evade
FNK per-build Cada compilación tiene un byte de clave diferente. Todos los strings de API quedan en bytes distintos en cada ejecutable. Firmas estáticas de strings en AV
PK per-build 16 bytes aleatorios cifran el tráfico. Dos capturas de tráfico de dos implants distintos no muestran ningún patrón en común. Firmas de tráfico de red (IDS/IPS)
Decoy IAT El IAT visible sólo contiene funciones de DLLs multimedia e interfaz gráfica, no DLLs de red o proceso. Análisis estático de importaciones
Subsistema GUI El ejecutable usa el subsistema -mwindows. No abre ventana de consola ni hereda stdin/stdout del proceso padre. Detección visual por usuario, heurísticas de consola
Mutex global Global\LCRI_INST previene múltiples instancias del mismo implant. El nombre es único y XOR'd en cada build. Comportamiento de self-replication
Padding .rodata Un bloque de ~3 KB de mensajes de error de Windows Update se embebe como string estático. Sube la entropía aparente de la sección de datos legibles. Análisis de entropía heurístico
HTTP sobre puerto estándar Tráfico en el puerto configurado (ej: 443 o 80) con User-Agent de Chrome. Pasa desapercibido en redes que no inspeccionan TLS. Filtrado de tráfico por puerto/protocolo
X-MID en cabecera HTTP El identificador de máquina va en una cabecera HTTP estándar-looking, no en el cuerpo. Análisis de cuerpo HTTP por IDS
⚠ Limitaciones: LIBRECRI no implementa cifrado asimétrico, no usa TLS real, no usa process hollowing ni DLL injection. Ante un EDR moderno con análisis de comportamiento en memoria (AMSI, ETW hooks), el implant sería detectado eventualmente. Estas técnicas se omiten intencionalmente para mantener el código auditable en un contexto educativo.

Análisis Defensivo

Comprender el comportamiento del implant permite diseñar estrategias de detección precisas. A continuación se listan los indicadores de compromiso más relevantes y las mitigaciones aplicables.

Indicadores de Compromiso (IOC)

CategoríaIndicadorValor / Patrón
Mutex Nombre de mutex global Global\LCRI_INST
Red User-Agent HTTP Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Red Cabecera personalizada X-MID: <hostname>
Red Endpoints GET /cmd · POST /res (HTTP/1.0)
Red Patrón de polling GET /cmd cada ~850ms sobre TCP persistente
Proceso IAT inusualmente pequeño Solo imports de comctl32, crypt32, version, winmm, imm32
Proceso Subsistema PE GUI (no CONSOLE) sin ventana visible
Comportamiento Creación de procesos hijo cmd.exe /c <comando> con pipes heredados
Keylogger SetWindowsHookEx con WH_KEYBOARD_LL Detectado por EDR como hooking de bajo nivel
Persistencia Registro HKCU\Software\Microsoft\Windows\CurrentVersion\Run\<nombre>

Detección

  • Network IDS (Suricata/Snort): regla para detectar GET /cmd HTTP/1.0 + cabecera X-MID:. El patrón de polling regular (~850ms) es también anómalo para tráfico legítimo.
  • EDR con análisis de comportamiento: SetWindowsHookExA con tipo WH_KEYBOARD_LL desde un proceso sin ventana visible es señal de alarma alta en plataformas como CrowdStrike Falcon o SentinelOne.
  • Análisis estático PE: IAT con sólo DLLs gráficas en un proceso que hace networking es altamente anómalo. Herramientas como YARA o Detect-It-Easy señalan este desbalance.
  • Monitorización de registro: Sysmon Event ID 13 detecta escrituras en HKCU\...\Run.
  • Análisis de tráfico: Zeek/Wireshark muestran conexiones HTTP/1.0 periódicas sin cuerpo (polling vacío). HTTP/1.0 sin TLS en puerto 443 es en sí mismo una anomalía.

Mitigaciones

  • Application Whitelisting (Windows Defender Application Control): bloquea la ejecución de ejecutables no firmados descargados de Internet.
  • AMSI + Script Block Logging: los comandos PowerShell del módulo de webcam y micrófono (Add-Type C#) son capturados por AMSI antes de compilarse.
  • Network Segmentation: los endpoints del implant no deberían poder alcanzar IPs externas directamente. Un proxy con inspección TLS interceptaría el tráfico.
  • SmartScreen + MOTW: archivos descargados de Internet tienen el Mark of the Web (Zone.Identifier) que activa advertencias de SmartScreen antes de la ejecución.
  • EDR con protección de hooks: plataformas como Microsoft Defender for Endpoint bloquean la instalación de keyboard hooks desde procesos no confiables.
10

Conclusiones

LIBRECRI demostró que es posible construir desde cero, con herramientas de código abierto y conocimiento fundamental de las APIs de Windows, un RAT funcional que implementa las mismas técnicas que se observan en familias de malware activas en el mundo real.

Las lecciones más importantes del proyecto:

  • La evasión de análisis estático es accesible: ocultar imports y XOR-codificar strings no requiere de rootkits ni exploits. El daño que puede hacer un binario "simple" en un entorno sin EDR moderno es considerable.
  • HTTP como canal C2 es difícil de bloquear: en cualquier red donde haya navegación web, el implant puede comunicarse. La inspección profunda de paquetes con contexto de comportamiento es necesaria, no sólo el filtrado por puerto.
  • Las defensas modernas (EDR, AMSI, WDAC) son efectivas: el keylogger con WH_KEYBOARD_LL y el uso de Add-Type para compilar C# en tiempo de ejecución son detectados por EDRs de nivel empresarial. Esto confirma la importancia de mantener los endpoints con protección actualizada.
  • La base de conocimiento ofensivo fortalece la defensa: comprender cómo el adversario construye y opera un RAT permite escribir reglas de detección precisas, entender las alertas de los EDRs y priorizar correctamente los riesgos en una postura de seguridad.

El proyecto quedó completamente funcional en laboratorio, logrando sesiones interactivas con captura de pantalla, keylogging, audio y video sobre un canal HTTP polling con cifrado por compilación.
